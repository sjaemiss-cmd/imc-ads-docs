# IMC 광고 조사 Docs Page 설계서

**작성일:** 2026-04-07  
**목표:** IMC 수업의 팀과제 전 범위를 하나의 웹 문서로 통합하고, 각 과제별로 최신 광고 사례를 중복 없이 정리한 GitHub Pages 배포용 docs page를 구축한다.

## 1. 문제 정의

현재 작업은 PPT별 요약 수준을 넘어, `과제별`로 광고 사례를 다시 조사하고 웹 문서로 재구성하는 것이다. 최종 산출물은 단순 발표 자료가 아니라 다음 요구를 동시에 만족해야 한다.

- 링크 하나로 접근 가능한 웹 페이지여야 한다.
- 좌측에는 `과제 탭`이 있어야 한다.
- 우측 본문은 `핵심개념설명 → 사례 → 구현된 개념 정리`의 서사 구조를 유지해야 한다.
- 비교형 과제는 한쪽만 채우지 않고 `양쪽 사례를 각각 1개씩` 포함해야 한다.
- 모든 광고는 서로 달라야 한다.
- 세종 eCampus 포럼 `id=318977`에 이미 사용된 광고와 중복되지 않아야 한다.
- 가능한 한 최신 광고를 우선 사용해야 한다.

이 요구를 만족하려면 단순 HTML 한 장이 아니라, `정적 SPA + 데이터 분리` 구조가 가장 적합하다.

- 포럼에서 확인된 사례와 유사한 소재가 의심되면 최종 채택 전에 수동 검증한다.

## 3. 최종 제품 형식

최종 제품은 `GitHub Pages`에 배포하는 정적 웹 앱이다.

### 3.1 추천 형식

`정적 SPA + JSON 데이터 분리`

선정 이유는 다음과 같다.

- 사용자는 링크 하나만 열면 된다.
- 좌측 과제 탭과 우측 광고 서브탭을 한 화면에서 자연스럽게 탐색할 수 있다.
- 특정 과제/광고로 바로 진입하는 딥링크를 만들 수 있다.
- 조사 데이터만 바꿔도 문서를 갱신할 수 있어 유지보수가 쉽다.
- GitHub Pages에 빌드 없이 바로 배포 가능하다.

### 3.2 비선정안

- 단일 HTML 한 장
  - 구현은 쉽지만 17개 광고 데이터를 하드코딩하면 유지보수가 급격히 나빠진다.
- 과제별 멀티페이지 docs
  - 문서형 느낌은 강하지만 탭 전환 UX와 링크 공유성이 떨어진다.

## 4. 정보구조

## 4.1 좌측 사이드바

좌측에는 강의 단위 라벨을 유지하면서, 실제 클릭 대상은 `과제 탭`으로 둔다.

예시 구조:

- 1. IMC
  - 실증주의 vs 해석주의
- 2. 동기와 관여도
  - 욕구기반 동기 vs 인지기반 동기
  - 고관여 vs 저관여
- 3. 감각과 지각
  - 오감 자극 광고
  - 자극물 자체의 주의 요인 광고
  - 게슈탈트 집단화 원리 광고
- 4. 기억
  - 의미적 부호화 광고
  - 절차기억(스크립트) 광고
  - 재인적 단서 광고
- 5. 학습
  - 고전적 조건화 광고
  - 고정비율 강화 광고

  - 경험적 위계 광고
  - 다속성 모형 광고
  - 주관적 규범 광고

## 4.2 우측 상단 광고 서브탭

현재 선택된 과제 안에서 광고를 전환하는 서브탭 영역을 둔다.

- 비교형 과제: 광고 2개 탭
- 단일형 과제: 광고 1개 탭

예시:

- `실증주의 사례`
- `해석주의 사례`

또는

- `광고 A`
- `광고 B`

표기명은 최종 조사 후 `브랜드 + 캠페인명`으로 고정한다.

## 4.3 우측 본문 서사

광고 상세 본문은 아래 순서를 고정한다.

1. 핵심 개념 설명
2. 광고 사례 설명
3. 구현된 개념 정리
4. 출처 및 메타데이터

각 광고 페이지는 반드시 아래 필드를 보여준다.

- 과제명
- 광고명/브랜드명
- 실행 시기
- 실행 매체
- 영상 링크
- 핵심 개념 설명
- 광고 장면/카피/오디오/비주얼 설명
- 해당 개념이 구현된 방식의 분석
- 중복 검토 메모

예상 필드:

- `checkedAt`

  - `discussionTitle`

## 6. 사용자 흐름

### 6.1 기본 흐름

1. 사용자가 링크를 열면 첫 과제 탭이 기본 선택된다.
2. 좌측 과제 탭을 클릭하면 해당 과제의 제목과 광고 서브탭이 갱신된다.
3. 상단 광고 서브탭을 클릭하면 본문이 해당 광고 분석으로 바뀐다.
4. URL 해시도 함께 바뀌어 특정 상태를 공유할 수 있다.

### 6.2 딥링크

해시 라우팅을 사용한다.

예시:

- `#/task/need-vs-cognitive/ad/need-case`
- `#/task/recognition-cue/ad/main`

이 방식이면 GitHub Pages에서 별도 서버 라우팅 없이 동작한다.

## 7. UI 원칙

### 7.1 시각 방향

- `docs page` 느낌을 유지한다.
- 발표 슬라이드처럼 한 번에 화면 전환하는 구조는 쓰지 않는다.
- 어두운 배경 기반의 정보밀도 높은 레이아웃을 유지하되, 문서 가독성을 우선한다.
- 좌측 네비게이션은 고정, 우측 본문은 스크롤 가능 구조로 둔다.

### 7.2 본문 톤

- 문장은 보고서형으로 짧고 명확하게 쓴다.
- 과도한 마케팅 문구보다 개념 분석 중심으로 쓴다.
- 모든 광고에 동일한 목차와 카드 구조를 적용한다.

### 7.3 모바일 대응

- 모바일에서는 좌측 사이드바를 접이식 드로어로 변경한다.
- 광고 서브탭은 가로 스크롤 가능 영역으로 처리한다.
- 본문 카드는 세로 스택으로 재배치한다.

## 8. 조사 원칙

### 8.1 최신성

- 기본적으로 `2025년~2026년` 집행 광고를 우선 채택한다.
- 해당 개념에 맞는 적절한 최신 광고가 없을 경우에만 예외를 허용한다.
- 예외를 사용할 경우 메모 필드에 이유를 남긴다.

### 8.2 중복 방지

- 모든 과제에서 광고는 서로 달라야 한다.
- 비교형 과제 내부 두 광고도 서로 달라야 한다.
- eCampus 게시글 blacklist와 일치하는 광고는 제외한다.

### 8.3 조사 서사

모든 광고는 같은 분석 템플릿으로 서술한다.

- `핵심 개념 설명`
  - 이 광고에서 봐야 할 이론/개념을 먼저 설명
- `사례`
  - 광고의 장면, 카피, 오디오, 비주얼, 모델, 매체 맥락 설명
- `구현된 개념 정리`
  - 위 요소가 해당 개념과 어떻게 정확히 연결되는지 분석

## 9. 배포 방식

배포 대상은 GitHub Pages다.

### 9.1 저장소 구조

사이트 파일은 GitHub Pages 운영을 위해 저장소 루트 배포 또는 `docs/` 배포 중 하나로 구성할 수 있다. 현재 저장소 상황을 고려할 때, 구현 단계에서는 아래 두 안 중 더 단순한 안을 선택한다.

- 안 A: 사이트를 `docs/`에 두고 GitHub Pages source를 `/docs`로 설정
- 안 B: 사이트를 루트에 두고 GitHub Pages source를 root로 설정

권장안은 `안 A`다.

이유:

- 현재 저장소에 PPT, 분석 메모, 도구 스크립트가 섞여 있어 웹 산출물을 `docs/` 아래로 격리하는 편이 관리가 쉽다.
- GitHub Pages 설정도 단순하다.

### 9.2 링크 전략

- 메인 링크 하나를 공유한다.
- 해시 라우팅으로 과제/광고별 직접 링크도 제공할 수 있다.

## 10. 구현 범위

이번 구현 범위는 아래까지 포함한다.

- eCampus 중복 광고 blacklist 데이터 생성
- 좌측 과제 탭 + 상단 광고 서브탭 + 본문 렌더링 UI
- 과제별 광고 데이터 구조 설계 및 적용
- GitHub Pages 정적 배포 가능 상태 구성

이번 구현 범위에서 제외한다.

- 백엔드 서버
- 관리자 CMS
- 실시간 DB
- 검색 기능
- 댓글 기능

## 11. 검증 기준

구현 완료 시 아래를 만족해야 한다.

- 좌측에 14개 과제 탭이 보인다.
- 비교형 과제는 광고 2개 서브탭을 가진다.
- 단일형 과제는 광고 1개 서브탭을 가진다.
- 각 광고 본문은 `핵심개념설명 → 사례 → 구현된 개념 정리` 구조를 따른다.
- 광고 데이터는 HTML에서 분리되어 있다.
- 해시 URL로 특정 과제/광고에 바로 접근할 수 있다.
- GitHub Pages에 정적으로 배포 가능하다.
- 중복 광고 blacklist를 데이터 파일로 확인할 수 있다.

## 12. 구현 추천 순서

1. `blacklist.json`과 과제 목록 구조를 먼저 고정한다.
2. 최소 데이터로 SPA 셸을 만든다.
3. 한 개 과제, 두 개 광고로 UI를 먼저 검증한다.
4. 전체 14개 탭과 17개 광고 데이터를 채운다.
5. GitHub Pages 배포 설정을 마무리한다.
